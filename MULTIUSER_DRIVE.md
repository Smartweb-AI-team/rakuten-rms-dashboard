# 共有Google Drive で 複数人運用するための設計指針

このツールは現在 **ローカル SQLite ファイル(`rms_ads.db`) を Google Drive 同期フォルダに置く** 構成です。
社内 5〜10名で運用しつつ、休暇・引継ぎ・コンサル参加に耐える運用ルールを以下に示します。

> ⚠️ 大原則：**同じ DB ファイルを同時に 2 人が「書き込む」と破損する。**
> 読み取りは安全だが、書き込み（＝データ取得＝バックフィル/補完）は **常にシングルライター**で。

---

## 1. ファイル分割（マルチショップ・マルチライター）

「店舗ごとに DB ファイルを分ける」設計に統一します。

```
G:\マイドライブ\マーケティングツール\
├─ shops\
│   ├─ 275374\rms_ads.db          ← 店舗ごと
│   ├─ 412901\rms_ads.db
│   └─ ...
├─ roster.json                   ← 担当者・引継ぎマップ（下記）
└─ exports\                       ← CSV/PDFのバックアップ
```

UI の **設定 > DB保存先** に各PC上の同期パスを設定すれば、店舗を切り替えるだけで対応可能。

---

## 2. roster.json — 担当者・代理・コンサル・引継ぎを一元管理

```json
{
  "shops": {
    "275374": {
      "name": "凛(りん)",
      "primary": "tanaka@company.co.jp",
      "deputies": ["sato@company.co.jp"],
      "viewers":  ["all-marketing@company.co.jp"],
      "consultants": [
        { "email": "ext@consult.jp", "expires": "2026-08-31", "scope": "viewer" }
      ],
      "handover": null
    },
    "412901": {
      "name": "ABC Store",
      "primary": "sato@company.co.jp",
      "handover": { "from": "tanaka@", "to": "sato@", "until": "2026-07-15" }
    }
  }
}
```

| フィールド | 役割 |
|---|---|
| `primary` | 通常時の取得担当（**ライター**は常に 1 名） |
| `deputies` | 副担当。休暇中だけ `primary` を一時的に上書き |
| `viewers` | 読み取り専用閲覧者 |
| `consultants[]` | 外部コンサル。`expires` で期限切れに自動失効、`scope` で権限 |
| `handover` | 引継ぎ中の `from→to` を明示。終了日まで両者が見られる |

このファイル自体も Drive 上に置き、**全員に共有**するが**編集できるのは管理者だけ**にしておきます（Drive の権限機能で）。

---

## 3. 運用ルール（シンプル＆実務的）

### 通常時
- 各店舗の **取得は primary が 1 日 1 回**。バックフィル/補完も primary のみ。
- 他の担当・コンサルは **読み取り専用**で大丈夫（DB は同期で受け取れる）。

### 休暇代替
1. 管理者が `roster.json` の `deputies` に追加（または `handover` を期限付きで設定）。
2. **代替者は自分の PC で同じ DB パスを開く**だけ。
3. 復帰後、roster を元に戻す。

### コンサルタント
- `consultants[]` に `expires` 付きで追加。**全店舗ではなく担当店舗のみ**共有(Drive のフォルダ権限)。
- 期限が切れたら Drive 側の共有を解除（root の自動失効はないので管理者が確認）。

### 退職・引継ぎ
1. `handover.until` 付きで 2 週間程度の重複期間を設ける。
2. 新担当(`to`)が引継ぎ後、`primary` を更新。
3. 旧担当の Drive 共有を解除。

### 同店舗・複数担当
- **書き込み(取得)は 1 人**に決め、それ以外は読み取り専用に統一。
- 「分析ダッシュボードを別々に見る」は全員可（読み取りは安全）。

---

## 4. 「同時取得」を物理的に防ぐ排他ロック

`G:\...\shops\<shop>\.lock` というロックファイルを作って取得中はそれを置く運用にする
（実装は将来追加。今は roster.json での「primary 1 名ルール」で十分）。

---

## 5. いつ Supabase/Postgres に移行するか

以下のいずれかに該当したら移行。スキーマはほぼそのまま流用可（`shop_id` キー、`user_shops` 追加、RLS で `auth.uid()` 紐付け）。

- 同時取得が必要（複数店舗を並列でリアルタイム）
- 店舗数が増え合計 DB が **数 GB を超える**
- Web 経由で外出先からも見たい
- 監査ログ・承認フローが必要

それまでは **Drive 共有 + roster.json + シングルライター運用**で十分に運用可能です。
